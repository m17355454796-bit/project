# myshell 重定向功能实现日志

今天主要继续完善了自己写的 `myshell` 项目，重点实现了 shell 中的基础重定向功能，包括输出重定向 `>`、追加重定向 `>>` 和输入重定向 `<`。

一开始我先定义了几个宏，用来描述当前命令是否存在重定向：

```c
#define REDIR_NONE 0
#define REDIR_INPUT 1
#define REDIR_OUTPUT 2
#define REDIR_APPEND 3
```

其中 `REDIR_NONE` 表示没有重定向，`REDIR_INPUT` 表示输入重定向 `<`，`REDIR_OUTPUT` 表示输出重定向 `>`，`REDIR_APPEND` 表示追加重定向 `>>`。这样在后面解析命令时，就可以通过一个全局变量 `status` 来记录当前命令属于哪一种重定向状态。

然后我写了 `redir()` 函数，用来扫描用户输入的命令行。它会从 `inputLine` 开始逐字符检查，如果遇到 `>`，就继续判断后面一个字符是不是 `>`，如果是两个连续的 `>`，就说明是追加重定向；如果只有一个 `>`，就说明是普通输出重定向。如果遇到 `<`，就说明是输入重定向。

在这个过程中，我用 `*start = '\0'` 把重定向符号前面的命令部分截断。比如输入：

```bash
ls > out.txt
```

原本整行是：

```text
ls > out.txt
```

当程序扫描到 `>` 时，会把 `>` 的位置改成 `\0`，这样前半部分就变成了：

```text
ls
```

后面的 `out.txt` 会通过 `getFilename()` 函数取出来，保存到 `filename` 数组中。这样后面 `cutLine(inputLine)` 再进行命令切分时，就只会切分真正的命令部分，不会把 `>` 和文件名也当成普通参数传给 `execvp`。

今天我对这个地方理解比较深的一点是：**重定向符号不是普通命令参数，而是 shell 自己要提前处理的语法。** 如果不把 `>` 和文件名从命令参数中剥离出去，执行 `ls > out.txt` 的时候，`ls` 会把 `>` 和 `out.txt` 当成普通参数，而不会真的发生重定向。

后面我又写了 `doRedir()` 函数，用来真正执行重定向。这个函数会根据 `status` 的值选择不同的 `open` 方式：

如果是普通输出重定向 `>`，使用：

```c
open(filename, O_CREAT | O_WRONLY | O_TRUNC, 0666);
```

这里 `O_CREAT` 表示文件不存在就创建，`O_WRONLY` 表示只写，`O_TRUNC` 表示如果文件已经存在，就先清空原来的内容。

如果是追加重定向 `>>`，使用：

```c
open(filename, O_CREAT | O_APPEND | O_WRONLY, 0666);
```

这里 `O_APPEND` 表示写入内容追加到文件末尾，而不是覆盖原来的内容。

如果是输入重定向 `<`，使用：

```c
open(filename, O_RDONLY);
```

因为输入重定向只是从文件中读取内容，所以只需要只读打开。

打开文件之后，再通过 `dup2()` 改变标准输入或标准输出的指向。比如输出重定向时：

```c
dup2(fd, 1);
```

意思是把标准输出 `1` 重定向到打开的文件。之后程序里本来应该输出到终端的内容，就会写入文件中。输入重定向时使用：

```c
dup2(fd, 0);
```

意思是把标准输入 `0` 改成从文件读取。

这里还有一个关键点：**重定向必须放在子进程中执行，而不能放在父进程的主循环中执行。**

一开始我差点把 `open + dup2 + close` 写在 `Bash()` 主循环里，这样会导致父进程自己的标准输出也被重定向。也就是说，如果执行一次：

```bash
ls > out.txt
```

后面 shell 自己的提示符也可能被写进 `out.txt`，终端上看起来就像 shell 没反应了一样。这个问题本质上是因为父进程是整个 shell 的主体，不能随便改它的标准输入输出。正确做法是在 `fork()` 后的子进程里先调用 `doRedir()`，再调用 `execvp()` 执行命令。

所以现在 `excute()` 函数里的逻辑是：

```c
pid_t pid = fork();

if(pid == 0)
{
    doRedir();
    execvp(argv[0], argv);
}
else
{
    waitpid(pid, &status, 0);
}
```

这样每次重定向只影响当前子进程，不会污染父进程的 shell 主循环。

今天最后我进行了测试：

```bash
ls > out.txt
cat out.txt
```

这个测试成功了，说明普通输出重定向 `>` 已经可以把 `ls` 的结果写入文件。

然后我又测试了：

```bash
date >> out.txt
cat < out.txt
```

这个也成功了，说明追加重定向 `>>` 和输入重定向 `<` 也基本跑通了。

目前这个 shell 已经完成了比较核心的一部分功能：

```text
1. 能显示自定义提示符
2. 能读取用户输入
3. 能用 strtok 切分命令和参数
4. 能通过 fork 创建子进程
5. 能通过 execvp 执行外部命令
6. 能通过 waitpid 回收子进程
7. 能处理 cd、env、echo $?、exit 这些简单内建命令
8. 能实现单个输入/输出/追加重定向
```

但是现在还没有完全结束，后面还有几个问题需要继续修：

第一，`doRedir()` 里面如果 `open` 或 `dup2` 失败，现在不能简单 `return`，因为这是在子进程中执行的。如果失败后 `return`，子进程可能还会继续执行 `execvp`，这就不合理。后面应该把这些错误处理改成：

```c
exit(1);
```

第二，`execvp()` 后面也要加错误处理。如果用户输入不存在的命令，比如：

```bash
abcd
```

那么 `execvp` 会失败，这时候应该：

```c
perror("execvp");
exit(1);
```

第三，`getFilename()` 现在还比较简单，如果输入：

```bash
ls > a.txt bbb
```

可能会把后面的内容拼到文件名里。后面应该只取第一个非空格字符串作为文件名。

第四，目前只支持单个重定向，比如：

```bash
ls > out.txt
cat < input.txt
```

还不支持这种同时存在输入和输出重定向的形式：

```bash
cat < input.txt > out.txt
```

这个可以后面再做，现在先把单重定向彻底稳定下来。

总的来说，今天这个 myshell 已经不是只会执行命令的半成品了，而是开始具备 shell 的核心行为了。尤其是今天把 `redir()`、`getFilename()`、`doRedir()` 和 `fork + execvp` 之间的关系串起来之后，我对 shell 处理重定向的流程理解更清楚了：

```text
读取命令
解析重定向
截断命令部分
提取文件名
切分 argv
fork 子进程
子进程 doRedir
子进程 execvp
父进程 waitpid
```

这条流程今天基本跑通了。下一步不急着加管道，先把错误处理、文件名解析、异常输入这些边角修好。先把这个简易 shell 的第一阶段收尾做稳，不然功能越堆越多，最后就变成一坨能跑但谁也不敢碰的代码垃圾。
